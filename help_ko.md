# 성능 트렌드 분석 가이드

## 1. 개요
성능 트렌드 분석 대시보드는 웹 애플리케이션의 장기적인 성능 추이를 시각화하고, 시스템 상태를 다각도로 분석하기 위한 도구입니다. 1년간의 데이터를 1시간 단위의 정밀도로 유지하여, 트렌드 파악 및 용량 계획(Capacity Planning)에 활용할 수 있습니다.

이 대시보드는 크게 세 가지 분석 레이어로 구성됩니다.

| 차트 | 분석 목적 | 주요 질문 |
|------|----------|----------|
| 메인 시계열 차트 | 장기 트렌드 파악 | "언제부터 느려졌는가?" |
| Overall 히트맵 (상관 분석) | 원인 규명 | "왜 느려졌는가?" |
| 요일 × 시간대 히트맵 (패턴 분석) | 반복 패턴 탐지 | "어느 시간에 반복적으로 느려지는가?" |

세 차트를 함께 활용하면 **"언제, 왜, 어떤 패턴으로"** 문제가 발생하는지 입체적으로 파악할 수 있습니다.

### 1.1 분석 대상 선택
- **계층형 도메인 선택**: 브레드크럼(Breadcrumb) 방식으로 도메인 그룹 및 개별 도메인을 탐색합니다. 각 단계를 클릭하면 하위 항목이 표시됩니다.
- **비지니스 / 인스턴스 전환**: "대상" 영역의 라디오 버튼으로 분석 단위를 전환할 수 있습니다.
  - **인스턴스 모드**: 개별 서버 인스턴스를 선택하여 분석합니다. 인스턴스를 선택하면 CPU/메모리 메트릭이 추가로 활성화됩니다.
  - **비지니스 모드**: 비지니스 단위로 묶인 논리적 그룹을 선택하여 분석합니다. 비지니스 모드에서는 CPU/메모리 메트릭이 비활성화됩니다.
- **자동 데이터 갱신**: 도메인, 인스턴스 또는 비지니스를 변경하면 별도의 조회 버튼 없이 모든 차트와 히트맵이 자동으로 갱신됩니다.

## 2. 메인 시계열 차트
애플리케이션 상단에 배치된 메인 차트는 선택된 메트릭의 시간 경과에 따른 변화를 확인하는 주요 지표입니다. 1년간의 데이터를 1일 단위 정밀도로 기록하여 추세 분석과 용량 계획에 도움을 줍니다.

### 2.1 각 지표(메트릭스)의 정의
- **응답 시간 Response Time (ms):** 요청 처리 완료까지 걸린 평균 시간. 사용자 경험과 직결되는 중요한 지표입니다.
- **TPS Transactions Per Second:** 1초당 평균 처리 건수. 시스템의 현재 처리량(Throughput)을 나타냅니다.
- **Hit수 Hits (Count):** 특정 시간대 내 총 요청 횟수. 시스템의 부하량(트래픽)을 측정합니다.
- **동시 사용자 Concurrent Users:** 시스템에 활성화된 세션/사용자 수.
- **에러율 Error Rate (%):** 전체 요청 건수 대비 에러 발생 건수의 비율. 수식: `(에러 건수 / 총 요청 건수) × 100`. 에러율 상승은 애플리케이션 버그나 인프라 장애를 알리는 중요한 신호입니다.
- **CPU / 메모리:** 개별 인스턴스의 물리적 리소스 사용률. 평균값과 최대값(Peak)을 함께 분석하여 리소스 부족을 조기에 감지할 수 있습니다. **인스턴스를 선택한 경우에만 활성화**됩니다.

### 2.2 대시보드 조작 방법
- **기간 범위 지정:** 메인 차트 상의 특정 범위를 마우스로 드래그하면, 해당 기간의 통계 데이터가 즉시 모든 카드와 히트맵에 반영됩니다.
- **범위 이동:** 키보드 방향키 (`←`, `→`)를 활용해 선택된 구간을 앞뒤로 부드럽게 이동시킬 수 있습니다.
- **범위 확장/축소:** `Shift` + 방향키(`←`, `→`)를 누르면 선택 범위의 시작 또는 끝 경계를 확장하거나 축소할 수 있습니다. 활성 경계(시작/끝)는 드래그 방향에 따라 자동으로 전환됩니다.
- **선택 해제:** `Esc` 키를 누르면 선택된 영역이 해제되며 1년 365일 전체 데이터로 되돌아갑니다.
- **이동 평균 활용:** '이동 평균' 토글을 켜면 단기적인 변동 노이즈가 제거된 장기 트렌드가 시각화되어 본질적인 추세를 쉽게 파악할 수 있습니다.

### 2.3 왜 이 차트가 중요한가?

> **핵심 한 줄 요약:** 메인 시계열 차트는 시스템의 "성적표"입니다. 일일 점수들이 쌓여 1년의 흐름을 만들고, 그 흐름을 보면 성능이 좋아지고 있는지, 나빠지고 있는지 한눈에 알 수 있습니다.

**장기 트렌드 시각화 — "큰 그림"을 본다**

날씨를 예로 들면, 오늘 하루가 유독 더웠다고 해서 "지구 온난화가 진행 중"이라고 단정할 수 없습니다. 1년치 기온 데이터를 쌓아 봐야 비로소 "여름이 해마다 길어지고 있다"는 구조적 변화를 읽을 수 있습니다. 성능도 마찬가지입니다.

메인 시계열 차트는 1년 단위의 데이터를 일별로 보여줌으로써, **월간·분기별 성능 추이와 계절성 트래픽 변동**(예: 연말 쇼핑 시즌, 특정 이벤트 주기)을 한눈에 파악할 수 있게 합니다. 일시적인 스파이크(일회성 급등)와 구조적인 성능 저하(서서히 나빠지는 추세)를 명확하게 구분할 수 있는 것이 가장 큰 강점입니다.

**용량 계획 (Capacity Planning) — "미래를 대비"한다**

병원에서 환자 수가 꾸준히 늘고 있다면, 의사와 병실을 미리 늘려야 합니다. 갑자기 환자가 몰린 날에야 대응하면 이미 늦은 것입니다. 서버 인프라도 동일합니다. 과거 TPS나 동시 사용자 추이를 보면 "3개월 후에는 현재 서버 용량의 한계에 다다를 것"이라는 예측이 가능합니다. 메인 차트의 데이터는 이처럼 **"언제쯤 인프라 증설이 필요한가"를 정량적으로 판단하는 기초 자료**가 됩니다.

**변경 영향도 검증 — "내 작업이 효과가 있었는지"를 확인한다**

코드를 배포하거나 서버 구성을 변경한 후, 성능이 실제로 나아졌는지 아니면 오히려 나빠졌는지 어떻게 알 수 있을까요? 메인 차트에서 배포 전·후 응답 시간과 에러율의 변화를 시계열로 보면 됩니다. 개선 효과가 명확하다면 배포 성공, 특정 지표가 악화됐다면 회귀 버그(Regression)가 발생한 것으로 즉시 진단할 수 있습니다.

**실무 활용 예시**

| 관찰 패턴 | 해석 | 권장 액션 |
|----------|------|----------|
| 응답 시간이 3개월간 완만하게 우상향 | 구조적 성능 저하 | DB 쿼리 최적화 또는 서버 증설 검토 |
| 특정 배포일 이후 에러율 급등 | 회귀 버그 의심 | 해당 배포 롤백 후 원인 분석 |
| 매년 연말에 TPS · Hit수 급증 | 계절성 트래픽 | 연말 전 사전 스케일아웃 계획 수립 |
| 이동 평균선은 안정적이나 원본선이 간헐적으로 급등 | 일회성 이벤트/장애 | 해당 날짜의 이벤트 로그 확인 |

### 2.4 이동 평균선 활용법
이동 평균(Moving Average)은 일별 변동 폭을 줄여 본질적인 진행 방향을 강조합니다. 마치 주식 차트에서 일별 등락보다 중기 추세선을 보는 것과 같습니다. 본 시스템은 **7일 이동 평균**과 **30일 이동 평균** 2가지를 지원합니다.

#### ▶ 7일 이동 평균 (단기 트렌드)
- 주 단위의 경향성 변화를 빠르게 포착합니다.
- 배포 직후 며칠 내 성능에 어떤 변화가 있었는지 점검하는 데 유용합니다.

#### ▶ 30일 이동 평균 (장기 트렌드)
- 월 단위의 근본적·구조적 트렌드를 확인합니다.
- 30일선이 지속해서 우상향(응답 시간, 에러율 기준)한다면, 스케일아웃이나 최적화 검토가 필요합니다.

#### ▶ 분석 포인트
- **7일선과 30일선의 이격:** 7일선이 30일선 위로 상승하기 시작하면 최근 성능 악화의 징조이며, 아래로 내려가기 시작하면 개선을 뜻합니다.
- **골든 크로스 / 데드 크로스 (응답 시간·에러율 기준):**
  - **데드 크로스:** 단기선(7일)이 장기선(30일)을 아래에서 위로 돌파 → 성능 악화 시그널. 빠른 원인 분석이 필요합니다.
  - **골든 크로스:** 단기선(7일)이 장기선(30일) 위에서 아래로 교차 → 성능 개선 시그널. 최근 튜닝이나 배포가 효과를 발휘하고 있다는 신호입니다.
- **원본 데이터와의 차이:** 이동 평균선 범주를 크게 벗어난 원본 데이터는 정상 범위에 속하지 않는 일회성 이벤트/장애일 확률이 높습니다. 해당 시점의 배포 기록이나 외부 이슈를 교차 확인하세요.

## 3. Overall 히트맵 분석 (상관 분석)
이 차트는 **부하(Hits)**와 **성능 지표** 간의 상관관계를 나타냅니다. 전체 데이터의 중앙값(Median)을 기준으로 4개 구역(Quadrant)으로 분류됩니다. 지표 중앙값은 1년 전체 주기를 기준으로, 부하 중앙값은 현재 범위 선택된 기간을 기준으로 산출됩니다.

- **좌측 상단: 저부하 × 고지표** 부하(Hits)가 낮음에도 불구하고 Y축 수치가 높은 상태.
- **우측 상단: 고부하 × 고지표** 부하(Hits)와 Y축 수치 모두 함께 높은 상태.
- **좌측 하단: 저부하 × 저지표** 부하(Hits)와 Y축 수치 모두 함께 낮은 상태.
- **우측 하단: 고부하 × 저지표** 부하(Hits)가 높음에도 불구하고 Y축 수치가 낮은 상태.

### 3.1 메트릭별 해석 가이드
Y축에 지정된 지표 성능에 따라 각 구역의 해석이 달라집니다.

#### ▶ 응답 시간 / 에러율 / CPU / 메모리 (값이 높을수록 악화)
- **좌상단 (⚠️):** 부하가 적은데 지표가 높음 → 슬로우 쿼리, 메모리 릭 등 애플리케이션 내부의 비효율 의심.
- **우상단 (🔴):** 부하도 많고 지표도 높음 → 수용 가능(Capacity) 한계 돌파 직전. 리소스 증설과 튜닝 급선무.
- **좌하단 (🟢):** 부하도 적고 지표도 낮음 → 가장 흔하게 관측되는 안정적이고 이상적인 상태.
- **우하단 (✅):** 부하가 몰렸는데 지표는 안정적 → 매우 효율적인 아키텍처. 인프라 여유분 충분 상태.

#### ▶ TPS (값이 높을수록 우수)
- **좌상단 (✅):** 적은 부하로 많은 처리 달성 → 매우 효율적인 로직 실행이 이뤄지고 있습니다.
- **우상단 (🟢):** 많은 부하를 빠르게 처리 → 시스템이 이상적인 최대 성능을 발휘 중입니다.
- **좌하단 (⚪):** 낮은 트래픽에 처리 건수도 적음 → 트래픽이 한산한 시간대(유휴)입니다.
- **우하단 (⚠️):** 요청 부하는 많으나 TPS가 떨어짐 → 심각한 병목 현상 발생. 처리가 원활하게 이루어지지 않음.

### 3.2 왜 이 차트가 중요한가?

> **핵심 한 줄 요약:** 메인 차트가 "언제 문제가 생겼는지"를 알려준다면, Overall 히트맵은 "그 문제가 트래픽 때문인지, 아니면 시스템 자체 결함인지"를 가려내는 탐정 역할을 합니다.

**부하와 성능 간 인과관계 추적 — "왜 느려졌는지"를 가린다**

메인 시계열 차트만 보면 "3월 15일에 응답 시간이 급등했다"는 사실은 알 수 있습니다. 하지만 그것이 사용자가 몰렸기 때문인지, 아니면 트래픽과 무관하게 특정 쿼리가 폭주했기 때문인지는 알 수 없습니다. 마치 교통 체증이 발생했을 때 "차가 많아서인지" 아니면 "신호등이 고장나서인지" 원인이 다른 것처럼, 대응 전략도 완전히 달라집니다.

Overall 히트맵은 동일 시간대의 부하(Hits)와 성능 지표를 한 점으로 동시에 표현합니다. 점이 **좌상단(저부하 × 고응답)**에 찍혀 있다면 트래픽은 적었지만 시스템 내부에서 문제가 발생한 것이고, **우상단(고부하 × 고응답)**이라면 트래픽 자체가 한계를 넘은 것입니다. 원인 진단 시간을 획기적으로 단축할 수 있습니다.

**시스템 한계 지점 도출 — "언제 터질지"를 미리 안다**

점들의 분포가 시간이 지날수록 우상단 쪽으로 이동하기 시작한다면, 현재 인프라 구조가 트래픽 증가를 더 이상 버티지 못할 임계점에 가까워지고 있다는 신호입니다. 이 추세를 조기에 감지하면 서비스 장애가 발생하기 **전에** 스케일아웃 등의 선제적 조치를 취할 수 있습니다.

**최적화 체감 검증 — "튜닝이 실제로 효과가 있었는지"를 확인한다**

인프라 교체나 코드 최적화를 진행했다면, "체감상 빨라진 것 같다"는 말 대신 데이터로 증명할 수 있어야 합니다. 튜닝 전 기간과 후 기간을 각각 선택해 히트맵 분포가 어떻게 이동했는지 비교하면, 개선 효과를 시각적으로 검증할 수 있습니다. 예를 들어, 기존에 우상단에 몰려 있던 점들이 최적화 후 우하단(고부하 × 저응답 = 효율적)으로 이동했다면 튜닝이 성공했음을 객관적으로 확인할 수 있습니다.

**실무 활용 예시**

| 히트맵 분포 패턴 | 해석 | 권장 액션 |
|---------------|------|----------|
| 점 대부분이 좌하단에 밀집 | 건강한 시스템 | 현 상태 유지, 정기 모니터링 |
| 점이 좌상단에 산발적으로 출현 | 내부 로직 비효율 | 슬로우 쿼리 분석, 메모리 릭 점검 |
| 점이 우상단으로 점차 이동 | 용량 한계 접근 | 스케일아웃 계획 수립 |
| 튜닝 후 점 분포가 우하단으로 이동 | 최적화 성공 | 개선 효과 확정, 보고서 작성 |

### 3.3 활용 팁
- **분포 편향도 검토:** 좌측 하단(안정)에 군집이 뭉쳐있다면 아주 건전한 시스템입니다. 우상단으로 넓게 파편이 퍼질수록 리스크가 높다는 분석이 됩니다.
- **기간 범위 지정과 연동 적용:** 메인 차트에서 특정 월 또는 주의 영역을 선택하면, 해당 기간 내 인과관계 분포 변화를 집중 분석해 계절적 특성을 교차 검토할 수 있습니다.
- **지표 전환 탐색:** 유사 기간에 메트릭만 바꿈으로써, "트래픽 쇄도로 인해 응답 지연은 발생했지만 에러는 발생하지 않고 버텼음"이라는 종합적 컨텍스트 확보가 가능합니다.
- **인스턴스 / 비지니스 분리 조회:** 특정 서버 인스턴스 또는 비지니스 단위로 대상을 한정하여 분포를 대조함으로써, 일부 서버만 노후화 문제를 겪고 있는지 또는 특정 비지니스에서만 문제가 발생하는지 확인할 수 있습니다.

## 4. 요일 × 시간대별 분석 (패턴 분석)
월~일요일, 그리고 0~23시 시간대 단위로 평균치를 조각내 히트맵 패턴으로 시각화한 차트입니다. 통계적 사분위수(IQR) 기준에 기반해 상태를 자동으로 감지해 분할합니다.

### 4.1 상태 기준 구분
- **안정 (Stable):** 과거 누적 통계를 기준으로 볼 때 표준 편차 내에 안착된 가장 정상적인 운영 구간 (Q3 75% 지점 이내).
- **경계 (Warning):** 평상시보다 지표가 두드러지게 벗어나기 시작한(Q3 ~ Q3 + 1.5 × IQR) 구간으로, 주의 깊은 모니터링이 필요합니다.
- **위험 (High):** 통계적 이상 수치 구간(Peak > Q3 + 1.5 × IQR 초과). 임계 전선에 다다랐거나 이례적인 트래픽 과밀 현상이 발생했습니다.

### 4.2 왜 이 차트가 중요한가?

> **핵심 한 줄 요약:** Overall 히트맵이 "왜 문제가 생겼는지"를 진단한다면, 요일 × 시간대 차트는 "그 문제가 매주 같은 시간에 반복되는가"를 밝혀내는 달력 역할을 합니다.

**주기적 패턴 발굴 — "예측 가능한 위험"을 사전에 막는다**

지하철 혼잡도는 무작위가 아닙니다. 평일 오전 8~9시, 오후 6~7시는 항상 붐빕니다. 서버 트래픽도 마찬가지입니다. "매주 목요일 오후 2시에 배치 작업이 몰려서 응답 시간이 급등한다"는 패턴이 있다면, 이는 예방 가능한 문제입니다. 요일 × 시간대 차트는 이처럼 반복되는 부하 패턴을 시각화하여, 사전에 리소스를 예약하거나 배치 작업 스케줄을 분산하는 등의 **선제적 대응**이 가능하도록 합니다.

**이상 탐지 가이드라인 — "평소와 다른 이상한 날"을 빠르게 잡아낸다**

이 차트의 진가는 **기대값의 정립**에 있습니다. 매주 화요일 오전 10시는 항상 파란색(안정)이어야 한다는 것을 차트가 학습합니다. 만약 어느 날 그 시간대에 갑자기 노란색(경계)이나 빨간색(위험) 셀이 나타난다면, 그것은 "평상시 사용자 패턴으로는 설명되지 않는 이상 현상"이 발생했다는 신호입니다. 임계치를 설정하고 모니터링하는 것보다 훨씬 직관적으로 이상 징후를 체감할 수 있습니다.

**자원 계획 수립 — "언제 서버를 늘리고, 언제 작업할지"를 계획한다**

트래픽 피크(최고점) 시간대를 정확히 파악하면 두 가지 실질적인 이점이 생깁니다. 첫째, **오토 스케일링(Auto-Scaling) 정책**을 더 정밀하게 설정할 수 있습니다. 트래픽이 몰리기 15~30분 전에 서버를 미리 증설하도록 규칙을 정의하면, 부하가 실제로 치솟는 시점에 이미 준비된 상태가 됩니다. 둘째, **서버 패치나 유지보수 작업 시간**을 트래픽이 가장 적은 안전한 시간대에 배치할 수 있습니다. 예를 들어, 차트에서 일요일 새벽 3시가 항상 안정(파란색)이라면 그 시간이 최적의 정기 점검 시간입니다.

**실무 활용 예시**

| 관찰 패턴 | 해석 | 권장 액션 |
|----------|------|----------|
| 매주 월요일 오전 9시에 빨간색 셀 출현 | 주간 업무 시작 집중 접속 | 해당 시간 전 Auto-Scaling 규칙 설정 |
| 평소 파란색인 시간대에 갑작스러운 노란색 출현 | 이상 트래픽 또는 장애 징후 | 해당 시점 로그 즉시 확인 |
| 특정 요일 전체 시간대가 빨간색 | 그 요일에 집중된 배치 작업 | 배치 스케줄 분산 검토 |
| 특정 시간대가 항상 파란색 | 최저 트래픽 구간 | 유지보수 작업 Maintenance Window로 지정 |

### 4.3 활용 팁
- **정기 유지보수 설정:** '안정' 빈도가 높은 구역(예: 일요일 새벽 3~5시)을 추려 가장 안전한 점검·정비(Maintenance Window) 시간표를 편성합니다.
- **오토 스케일링 준비성:** '경계' 이상 구간이 집중되는 요일·시간대에 맞춰 오토 스케일 업 규칙을 정의해 정책을 보완합니다.
- **여러 지표 전환 연속 비교:** 대시보드 상단에서 응답 시간 → 에러율 → CPU 순으로 메트릭을 순환하며 교차 분석하여 장애 파이프라인과 구체적인 병목(Bottle-Neck) 구간을 찾아냅니다.
- **기간 범위 지정 융합 적용:** 메인 차트에서 특정 이벤트 기간이나 분기를 선택하면, 해당 기간만의 요일별 패턴 상관 분석이 가능해져 계절적 특성을 파악하는 데 유리합니다.

## 5. 세 차트를 함께 쓰는 법 — 문제 진단 워크플로우

실제 장애 또는 성능 이상이 감지됐을 때, 아래 순서로 세 차트를 순차적으로 활용하면 원인을 체계적으로 좁혀나갈 수 있습니다.

```
[1단계] 메인 시계열 차트
  → "언제부터 응답 시간이 올라갔는가?" 이상 발생 시점과 기간을 특정

[2단계] Overall 히트맵 (상관 분석)
  → "트래픽이 많아서인가, 아니면 시스템 내부 문제인가?" 원인 유형 분류

[3단계] 요일 × 시간대 히트맵 (패턴 분석)
  → "이 패턴이 반복되는가?" 일회성 장애인지, 구조적 반복 문제인지 판단

[결론] 원인 유형별 대응
  - 트래픽 초과 → 스케일아웃 / Auto-Scaling 정책 보완
  - 내부 비효율 → 슬로우 쿼리, 메모리 릭, 코드 최적화
  - 반복 패턴 → 배치 스케줄 분산 / 피크 시간 선제 리소스 확보
```
