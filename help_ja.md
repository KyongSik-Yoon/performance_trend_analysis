# 性能トレンド分析 ガイド

## 1. 概要
性能トレンド分析ダッシュボードは、ウェブアプリケーションの長期的な性能推移を可視化し、システムの状態を多角的に分析するためのツールです。1年間のデータを1時間単位の精度で保持し、トレンドの把握やキャパシティプランニングに活用できます。

本ダッシュボードは、大きく3つの分析レイヤーで構成されています。

| チャート | 分析目的 | 主な問い |
|---------|---------|---------|
| メイン時系列チャート | 長期トレンドの把握 | 「いつから遅くなったのか？」 |
| Overall ヒートマップ（相関分析） | 原因の究明 | 「なぜ遅くなったのか？」 |
| 曜日 × 時間帯 ヒートマップ（パターン分析） | 反復パターンの検出 | 「どの時間帯に繰り返し遅くなるのか？」 |

3つのチャートを組み合わせることで、**「いつ、なぜ、どのようなパターンで」** 問題が発生しているかを立体的に把握できます。

### 1.1 分析対象の選択
- **階層型ドメイン選択**: ブレッドクラム（Breadcrumb）方式でドメイングループおよび個別ドメインを階層的に探索できます。各段階をクリックすると下位項目が表示されます。
- **ビジネス / インスタンス切替**: 「対象」エリアのラジオボタンで分析単位を切り替えることができます。
  - **インスタンスモード**: 個別サーバーインスタンスを選択して分析します。インスタンスを選択するとCPU/メモリメトリクスが追加で有効化されます。
  - **ビジネスモード**: ビジネス単位でグルーピングされた論理グループを選択して分析します。ビジネスモードではCPU/メモリメトリクスは無効化されます。
- **自動データ更新**: ドメイン、インスタンス、またはビジネスを変更すると、別途の検索ボタンなしですべてのチャートとヒートマップが自動的に更新されます。

## 2. メイン時系列チャート
アプリケーション上部に配置されたメインチャートは、選択したメトリクスの時間経過に伴う変化を確認するための主要な指標です。1年間のデータを1日単位の精度で記録し、トレンド分析とキャパシティプランニングに役立ちます。

### 2.1 各指標（メトリクス）の定義
- **応答時間 Response Time (ms):** リクエストの処理完了までにかかった平均時間。ユーザー体験に直結する重要な指標です。
- **TPS Transactions Per Second:** 1秒あたりの平均処理件数。システムの現在のスループットを示します。
- **Hit数 Hits (Count):** 特定の時間枠内の総リクエスト数。システムの負荷量（トラフィック）を測定します。
- **同時ユーザ数 Concurrent Users:** システムにアクティブに接続しているユーザー数。
- **エラー率 Error Rate (%):** 総リクエスト数に対するエラー発生数の割合。計算式: `(エラー件数 / 総リクエスト数) × 100`。エラー率の上昇は、アプリケーションの不具合やインフラ障害を示す重要なシグナルです。
- **CPU / メモリ:** インスタンスごとの物理リソース使用率。平均値と最大値(Peak)を同時に分析することで、リソース不足を早期に検知できます。**インスタンスを選択した場合にのみ有効化**されます。

### 2.2 ダッシュボードの操作方法
- **期間の絞り込み:** メインチャート上の特定の範囲をマウスでドラッグすると、その期間の統計データが全カードとヒートマップに即座に反映されます。
- **範囲の移動:** キーボードの方向キー（`←`、`→`）を押すと、選択した範囲を前後にスムーズに移動できます。
- **範囲の拡張/縮小:** `Shift` + 方向キー（`←`、`→`）を押すと、選択範囲の開始または終了境界を拡張・縮小できます。アクティブな境界（開始/終了）はドラッグ方向に応じて自動的に切り替わります。
- **選択解除:** `Esc` キーを押すと、選択がリセットされ、1年全体のデータに戻ります。
- **移動平均の活用:** 「移動平均」をOnにすると、短期的な変動ノイズを除去した長期トレンドが視覚化され、本質的な変化を捉えやすくなります。

### 2.3 なぜこのチャートが重要か？

> **一行まとめ:** メイン時系列チャートはシステムの「成績表」です。日々のスコアが積み重なって1年の流れを作り、その流れを見ればパフォーマンスが良くなっているのか、悪くなっているのかが一目で分かります。

**長期トレンドの可視化 ―「全体像」を見る**

天気を例にすると、今日一日が特に暑かったからといって「地球温暖化が進行中」と断定することはできません。1年分の気温データを蓄積してこそ「夏が年々長くなっている」という構造的な変化を読み取ることができます。パフォーマンスも同様です。

メイン時系列チャートは1年分のデータを日単位で表示することで、**月次・四半期ごとの性能推移や季節性トラフィック変動**（例：年末セール期間、特定イベント周期）を一目で把握できます。一時的なスパイク（単発的な急上昇）と構造的な性能劣化（徐々に悪化する傾向）を明確に区別できることが最大の強みです。

**キャパシティプランニング ―「未来に備える」**

病院で患者数が着実に増加しているなら、医師や病室を事前に増やす必要があります。患者が殺到してから対応しては手遅れです。サーバーインフラも同じです。過去のTPSや同時接続ユーザーの推移を見れば、「3か月後には現在のサーバー容量の限界に達するだろう」という予測が可能です。メインチャートのデータはこのように**「いつ頃インフラ増設が必要か」を定量的に判断するための基礎資料**となります。

**変更影響の検証 ―「自分の作業に効果があったか」を確認する**

コードをデプロイしたりサーバー構成を変更した後、パフォーマンスが実際に改善したのか、それとも悪化したのか、どうすれば分かるでしょうか？メインチャートでデプロイ前後の応答時間やエラー率の変化を時系列で確認すればよいのです。改善効果が明確であればデプロイ成功、特定の指標が悪化していればリグレッションバグが発生したと即座に診断できます。

**実務活用例**

| 観察パターン | 解釈 | 推奨アクション |
|------------|------|-------------|
| 応答時間が3か月間緩やかに右肩上がり | 構造的な性能劣化 | DBクエリ最適化またはサーバー増設を検討 |
| 特定デプロイ日以降エラー率が急上昇 | リグレッションバグの疑い | 該当デプロイのロールバック後に原因分析 |
| 毎年年末にTPS・Hit数が急増 | 季節性トラフィック | 年末前に事前スケールアウト計画を策定 |
| 移動平均線は安定しているが原データが断続的に急騰 | 単発イベント/障害 | 該当日のイベントログを確認 |

### 2.4 移動平均線の活用方法
移動平均(Moving Average)は日次の変動ノイズを除去し、本質的なトレンドを浮き彫りにします。まるで株価チャートで日々の騰落より中期トレンドラインを見るのと同じです。本ツールでは **7日移動平均** と **30日移動平均** の2種類を提供しています。

#### ▶ 7日移動平均（短期トレンド）
- 週単位の傾向変化を素早く検出できます。
- デプロイ後の性能変化を数日以内に確認する際に有効です。

#### ▶ 30日移動平均（長期トレンド）
- 月単位の根本的・構造的トレンドを把握できます。
- 30日線が持続的に上昇（応答時間・エラー率の場合）している場合、スケーリングやチューニングの検討が必要です。

#### ▶ 分析のポイント
- **7日線と30日線の乖離:** 7日線が30日線を上回り始めたら直近の性能悪化の兆候、下回り始めたら改善の兆候です。
- **ゴールデンクロス / デッドクロス（応答時間・エラー率基準）:**
  - **デッドクロス:** 短期線（7日）が長期線（30日）を下から上に突き抜ける → 性能悪化シグナル。速やかな原因分析が必要です。
  - **ゴールデンクロス:** 短期線（7日）が長期線（30日）の上から下に交差 → 性能改善シグナル。最近のチューニングやデプロイが効果を発揮している兆候です。
- **原データとの差:** 原データが移動平均線から大きく乖離した日は、一時的なイベントや障害を示す外れ値である可能性が高いです。該当時点のデプロイ記録や外部イシューをクロスチェックしてください。

## 3. Overall ヒートマップ分析（相関分析）
このチャートは、**負荷(Hits)**と**性能指標**の相関を示します。データは中央値(Median)によって4つのエリアに分類されます。性能指標の中央値は全体期間を基準に、Hit数の中央値は選択した期間を基準に算出されます。

- **左上: 低負荷 × 高指標値** 負荷(Hits)が少ないにもかかわらず、Y軸の指標値が高い状態。
- **右上: 高負荷 × 高指標値** 負荷(Hits)とY軸の指標値が共に高い状態。
- **左下: 低負荷 × 低指標値** 負荷(Hits)とY軸の指標値が共に低い状態。
- **右下: 高負荷 × 低指標値** 負荷(Hits)が高いにもかかわらず、Y軸の指標値が低い状態。

### 3.1 メトリクス別の解析ガイド
Y軸に表示される指標の種類によって、各エリアの意味が変わります。

#### ▶ 応答時間 / エラー率 / CPU / メモリ（値が高い＝悪化）
- **左上（⚠️）:** 低負荷でも指標が高い → アプリケーション内部の非効率（遅いクエリ、メモリリーク等）が疑われます。
- **右上（🔴）:** 高負荷で指標も高い → キャパシティの限界。リソースの増設やチューニングが急務です。
- **左下（🟢）:** 低負荷で指標も低い → 安定した理想的な状態です。
- **右下（✅）:** 高負荷でも指標が低い → 非常に効率的なアーキテクチャ。インフラに十分な余裕があります。

#### ▶ TPS（値が高い＝良好）
- **左上（✅）:** 低負荷で高スループット → 効率的な処理が行われています。
- **右上（🟢）:** 高負荷で高スループット → 最適なパフォーマンス状態です。
- **左下（⚪）:** 低負荷で低スループット → アイドル状態またはトラフィックが少ない時間帯です。
- **右下（⚠️）:** 高負荷で低スループット → ボトルネックの可能性。処理が追いついていません。

### 3.2 なぜこのチャートが重要か？

> **一行まとめ:** メインチャートが「いつ問題が発生したか」を教えてくれるなら、Overallヒートマップは「その問題がトラフィックのせいか、システム自体の欠陥か」を見極める探偵の役割を果たします。

**負荷と性能の因果関係の追跡 ―「なぜ遅くなったか」を見分ける**

メイン時系列チャートだけを見ると「3月15日に応答時間が急上昇した」という事実は分かります。しかし、それがユーザーが殺到したためなのか、トラフィックとは無関係に特定のクエリが暴走したためなのかは分かりません。まるで交通渋滞が発生した時に「車が多いからか」それとも「信号機が故障したからか」で原因が異なるように、対応戦略もまったく変わってきます。

Overallヒートマップは同一時間帯の負荷(Hits)と性能指標を1つの点として同時に表現します。点が**左上（低負荷 × 高応答）**にプロットされていればトラフィックは少なかったがシステム内部で問題が発生したことを意味し、**右上（高負荷 × 高応答）**であればトラフィック自体が限界を超えたことを意味します。原因診断の時間を飛躍的に短縮できます。

**システム限界点の導出 ―「いつ破綻するか」を事前に知る**

データポイントの分布が時間とともに右上方向に移動し始めたら、現在のインフラ構成がトラフィック増加にこれ以上耐えられない臨界点に近づいているという信号です。この傾向を早期に検知すれば、サービス障害が発生する**前に**スケールアウトなどの先制的措置を講じることができます。

**最適化効果の検証 ―「チューニングが実際に効果があったか」を確認する**

インフラの入れ替えやコード最適化を行ったなら、「体感的に速くなった気がする」という言葉ではなくデータで証明できなければなりません。チューニング前の期間と後の期間をそれぞれ選択してヒートマップの分布がどう移動したかを比較すれば、改善効果を視覚的に検証できます。例えば、以前は右上に集中していたデータポイントが最適化後に右下（高負荷 × 低応答 = 効率的）に移動していれば、チューニングが成功したことを客観的に確認できます。

**実務活用例**

| ヒートマップ分布パターン | 解釈 | 推奨アクション |
|----------------------|------|-------------|
| ほとんどの点が左下に密集 | 健全なシステム | 現状維持、定期モニタリング |
| 点が左上に散発的に出現 | 内部ロジックの非効率 | スロークエリ分析、メモリリーク点検 |
| 点が右上方向に徐々に移動 | 容量限界に接近 | スケールアウト計画の策定 |
| チューニング後に点の分布が右下に移動 | 最適化成功 | 改善効果の確定、レポート作成 |

### 3.3 活用ガイド
- **分布の偏りを確認:** 左下（安定）にデータが密集していれば健全な状態。右上へ広がるほどキャパシティリスクが高まっています。
- **期間絞り込みとの併用:** メインチャートで特定の月や週の範囲を選択すると、その期間内の因果関係の分布変化を集中的に分析し、季節的な特性をクロスチェックできます。
- **メトリクス切替による多面的分析:** 同じ期間でメトリクスを切り替えることで、「トラフィック殺到で応答遅延は発生したがエラーは出ずに耐えた」という総合的なコンテキストの把握が可能です。
- **インスタンス/ビジネス別比較:** 特定のサーバーインスタンスやビジネス単位に対象を限定して分布を比較することで、一部のサーバーだけが老朽化問題を抱えていないか、または特定のビジネスでのみ問題が発生していないかを確認できます。

## 4. 曜日 × 時間帯別分析（パターン分析）
月〜日曜日、0〜23時の時間帯単位で平均値をヒートマップパターンとして可視化したチャートです。統計的な四分位範囲(IQR)に基づき、ステータスを自動的に分類します。

### 4.1 ステータスの定義
- **安定 (Stable):** 過去の累積統計を基準として標準偏差内に収まった最も正常な運用区間（Q3 75%地点以下）。
- **警戒 (Warning):** 通常よりも指標が顕著に逸脱し始めた区間（Q3 〜 Q3 + 1.5 × IQR）で、注意深い監視が必要です。
- **高 (High):** 統計的異常値区間（Peak > Q3 + 1.5 × IQR超過）。リソースの限界に近い、またはイレギュラーなトラフィック過密現象が発生しています。

### 4.2 なぜこのチャートが重要か？

> **一行まとめ:** Overallヒートマップが「なぜ問題が発生したか」を診断するなら、曜日 × 時間帯チャートは「その問題が毎週同じ時間に繰り返されているか」を明らかにするカレンダーの役割を果たします。

**周期的パターンの発見 ―「予測可能な危険」を事前に防ぐ**

地下鉄の混雑度はランダムではありません。平日の朝8〜9時、夕方6〜7時は必ず混み合います。サーバートラフィックも同様です。「毎週木曜日の午後2時にバッチ処理が集中して応答時間が急上昇する」というパターンがあれば、それは予防可能な問題です。曜日 × 時間帯チャートはこのような反復的な負荷パターンを可視化し、事前にリソースを予約したりバッチスケジュールを分散させるなどの**先制的な対応**を可能にします。

**異常検知のガイドライン ―「普段と違う異常な日」を素早くキャッチする**

このチャートの真価は**期待値の確立**にあります。毎週火曜日の午前10時は常に青色（安定）であるべきだということをチャートが示しています。もしある日その時間帯に突然黄色（警戒）や赤色（高）のセルが現れたなら、それは「通常のユーザーパターンでは説明できない異常現象」が発生したという信号です。閾値を設定してモニタリングするよりもはるかに直感的に異常の兆候を察知できます。

**リソース計画の策定 ―「いつサーバーを増やし、いつ作業するか」を計画する**

トラフィックピーク（最高点）の時間帯を正確に把握すると、2つの実質的なメリットが生まれます。第一に、**オートスケーリング(Auto-Scaling)ポリシー**をより精密に設定できます。トラフィックが集中する15〜30分前にサーバーを事前増設するようルールを定義すれば、負荷が実際に急上昇する時点にはすでに準備完了の状態になります。第二に、**サーバーパッチやメンテナンス作業の時間**をトラフィックが最も少ない安全な時間帯に配置できます。例えば、チャートで日曜日の深夜3時が常に安定（青色）であれば、その時間が最適な定期点検時間です。

**実務活用例**

| 観察パターン | 解釈 | 推奨アクション |
|------------|------|-------------|
| 毎週月曜日の午前9時に赤色セル出現 | 週初め業務開始の集中アクセス | 該当時間前にAuto-Scalingルールを設定 |
| 普段青色の時間帯に突然黄色が出現 | 異常トラフィックまたは障害の兆候 | 該当時点のログを即座に確認 |
| 特定曜日の全時間帯が赤色 | その曜日に集中したバッチ処理 | バッチスケジュールの分散を検討 |
| 特定時間帯が常に青色 | 最低トラフィック区間 | メンテナンス作業のMaintenance Windowに指定 |

### 4.3 活用ガイド
- **定期メンテナンスの設定:** 「安定」の頻度が高い区間（例：日曜日の深夜3〜5時）を抽出し、最も安全な点検・整備（Maintenance Window）のスケジュールを策定します。
- **オートスケーリングの準備:** 「警戒」以上が集中する曜日・時間帯に合わせてオートスケールアップルールを定義し、ポリシーを補完します。
- **複数指標の連続切替比較:** ダッシュボード上部で応答時間 → エラー率 → CPU の順にメトリクスを循環させてクロス分析し、障害パイプラインと具体的なボトルネック区間を特定します。
- **期間絞り込みとの統合活用:** メインチャートで特定のイベント期間や四半期を選択すると、その期間だけの曜日別パターンの相関分析が可能になり、季節的な特性の把握に有利です。

## 5. 3つのチャートを併用する方法 ― 問題診断ワークフロー

実際の障害やパフォーマンス異常が検知された際、以下の順序で3つのチャートを段階的に活用すれば、原因を体系的に絞り込むことができます。

```
[ステップ1] メイン時系列チャート
  → 「いつから応答時間が上昇したか？」 異常発生の時点と期間を特定

[ステップ2] Overall ヒートマップ（相関分析）
  → 「トラフィックが多かったからか、それともシステム内部の問題か？」 原因タイプを分類

[ステップ3] 曜日 × 時間帯 ヒートマップ（パターン分析）
  → 「このパターンは繰り返されているか？」 一回限りの障害か、構造的な反復問題かを判断

[結論] 原因タイプ別の対応
  - トラフィック超過 → スケールアウト / Auto-Scalingポリシーの補完
  - 内部非効率 → スロークエリ、メモリリーク、コード最適化
  - 反復パターン → バッチスケジュールの分散 / ピーク時間の先制リソース確保
```
